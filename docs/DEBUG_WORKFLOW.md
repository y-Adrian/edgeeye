# Agent 协作调试交互规范

> 适用于 Milk-V Duo S 摄像头等项目调试。主工作文件：`docs/DEBUG_SESSION/`（按 ISSUE 拆分）  
> 静态背景文档：`docs/CAMERA_VERIFY.md`、`docs/SINGLE_CAM_BRINGUP.md`

> **v2 相对 v1 的改动摘要**  
>
> 1. 新对话强制上传文件，防止 AI 凭记忆分析（§0）  
> 2. Session 文件按 ISSUE 拆分 + INDEX 索引，避免主文件无限膨胀（§3、§10）  
> 3. 通过标准从「建议」升级为**必填**，Agent 不得在通过标准为空时标可执行（§7）  
> 4. 异常结果处理专节，覆盖超时/掉线/输出不完整等（§5.5）  
> 5. 触发语收敛为统一格式，消除歧义（§6）  
> 6. **命令与结果合并为同一代码块**（§5.1），粘贴时无需上下翻找（§5.4）  
> 7. **变更记录日期须从互联网获取**（§12），禁止硬编码或沿用旧日期

---

## 0. 新对话启动协议

> **每次打开新对话，必须先完成以下两步，否则 Agent 不得开始任何分析。**

### 0.1 用户必做

1. 上传 `docs/DEBUG_SESSION/INDEX.md`（ISSUE 索引与状态总览）。
2. 上传当前活跃 ISSUE 的文件，如 `DEBUG_SESSION_ISSUE003.md`。
3. 发送标准触发语（见 §6）启动分析。

### 0.2 Agent 必做

1. 确认收到上述文件后，才开始读取和分析。
2. 若用户未上传文件，直接回复：

   > 「请先上传 `INDEX.md` 和当前 ISSUE 文件，我才能开始分析，避免凭记忆出错。」

3. 不得根据聊天历史推测当前状态。

---

## 1. 核心理念

- **一个 issue = 一个独立文件**（如 `DEBUG_SESSION_ISSUE001.md`）。
- **一次只推进一步**；某步失败或卡住时，不继续后面预设步骤。
- **先提问 → 再执行 → 后分析**，避免在聊天里堆 1、2、3、4、5 步，执行到一半后面步骤全部失效。
- **长日志写在 markdown 里**，聊天里只发短触发语。
- **通过标准必填**；没有明确通过标准的步骤，Agent 不标「✅ 可以执行了」。

---

## 2. 单步生命周期

```text
⏳ 待执行  →  [执行前问答]  →  ✅ 可以执行了  →  [结果]  →  ✅ 已完成 + Agent 分析  →  下一步
              ↑ 你可先提问              ↑ 你动手跑命令
                                                  ↑ 结果异常时见 §5.5
```

| 阶段               | 用户                                                         | Agent                                                        |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **执行前**         | 在当步「执行前问答」或 ISSUE 通用问答里写疑问，**不必先跑命令** | 读 markdown → 在「Agent 解答」里回复 → **确认通过标准已填写** → 必要时改命令/备注 → 标 **✅ 可以执行了** |
| **执行后（正常）** | 在当步**唯一** `shell` 块中，`# --- 结果 ---` 下方粘贴输出（见 §5.4） | 读该块 → 对照通过标准判定 → 写「Agent 分析」→ 勾选完成 → **只追加下一步** |
| **执行后（异常）** | 在 `# --- 结论 ---` 按 §5.5 写明异常类型                       | 读异常结果 → 判断是否需要补充信息 → 给出处置建议，不强行推进下一步 |

## 3. 文档结构

### 3.1 文件组织（改为按 ISSUE 拆分）

```
docs/DEBUG_SESSION/
├── INDEX.md                        ← ISSUE 索引与状态总览（每次上传这个）
├── DEBUG_SESSION_ISSUE001.md       ← 已解决，归档
├── DEBUG_SESSION_ISSUE002.md       ← 已解决，归档
└── DEBUG_SESSION_ISSUE003.md       ← 当前活跃，每次上传这个
```

**INDEX.md 格式：**

```markdown
# DEBUG SESSION 索引

| ISSUE | 标题 | 状态 | 文件 |
|-------|------|------|------|
| ISSUE-001 | I2C 扫不到传感器 | ✅ 已解决 | DEBUG_SESSION_ISSUE001.md |
| ISSUE-002 | RTSP 无码率 | ✅ 已解决 | DEBUG_SESSION_ISSUE002.md |
| ISSUE-003 | 双摄切换花屏 | 🔄 进行中 | DEBUG_SESSION_ISSUE003.md |

**当前活跃：** ISSUE-003，Step 2 ⏳ 待执行
```

> 好处：每次只上传 INDEX + 当前 ISSUE 文件，AI 上下文窗口压力大幅降低；旧 ISSUE 自然归档。

### 3.2 单个 ISSUE 文件结构

```markdown
## ISSUE-00N: 标题

**状态：** 进行中 / 硬件待查 / 已解决 …

### ISSUE-00N 通用问答        ← 跨步骤的问题
**问题（用户填写）：**
**Agent 解答：**

### 已知事实（Agent 维护）     ← 已确认结论，勿随意删

### Step 1 — 标题 ✅ 已完成
#### 执行前问答
**状态：** ✅ 可以执行了
**问题（用户填写）：**
**Agent 解答：**
**通过标准：** ✅ 已填（示例：i2cdetect 在 0x36 位置出现地址）

```shell
# --- 命令 ---
（Agent 写入待执行命令）

# --- 结果（执行后原样粘贴在下方）---


# --- 结论 ---
```

**Agent 分析：**

### Step 2 — 标题 ⏳ 待执行   ← 当前只做标记为待执行的这一步
…
```

## 4. 执行前问答

### 4.1 两个问答位置

| 位置                   | 用途                          | 示例                                             |
| ---------------------- | ----------------------------- | ------------------------------------------------ |
| **ISSUE 通用问答**     | 整个 issue 相关、不限于某一步 | 「Step 5 测完怎么恢复双模组接线？」              |
| **每步「执行前问答」** | 只针对当前这一步的命令/操作   | 「J2 口金手指朝哪边？」「临时 ini 测完要删吗？」 |

### 4.2 填写方式

- 在 **「问题（用户填写）」** 下直接写，可编号（1. 2. 3.），可随时追加。
- **拿不准是否该执行时，先填这里，不要先动手。**
- Agent 解答后：
  - 在 **「Agent 解答」** 里写清回复；
  - **确认「通过标准」已填写**，若为空则先补填再继续；
  - 把当步状态从 **🟡 可提问** 改为 **✅ 可以执行了**；
  - 若需改命令，会同步改上方命令块并在解答里说明。

---

## 5. 结果填写约定

### 5.1 单块布局（命令 + 结果在一起）

每步只有 **一个** ` ```shell ` 代码块，自上而下三段，**避免命令在上、结果在远处下方来回翻**：

```shell
# --- 命令 ---
rmmod debris 2>/dev/null
ln -sf /mnt/data/sensor_cfg_GC2083.ini /mnt/data/sensor_cfg.ini
i2cdetect -y 3

# --- 结果（执行后原样粘贴在下方，可保留 [root@...]# 提示符）---
[root@milkv-duo]~# i2cdetect -y 3
     0  1  2  3 ...
30: -- -- -- -- -- -- -- 37

# --- 结论 ---
未通过
```

- Agent 写步骤时：**只建这一个块**，在 `# --- 结果 ---` 下留空行等你粘贴。
- **禁止**再单独写「板上/Mac 命令」+ 远处「结果（用户填写）」两个分离结构（旧版 v2 初稿已废弃）。

### 5.2 多终端 / Mac

仍在**同一块**内，用子注释分隔，紧接在对应命令段落后：

```shell
# --- 命令：板上 ---
netstat -ln | grep 8554

# --- 结果：板上 ---


# --- 命令：Mac ---
ffprobe -show_entries stream=bit_rate ...

# --- 结果：Mac ---


# --- 结论 ---
通过
```

### 5.3 正常情况注意

- **不必**在聊天里重复贴长日志。
- `# --- 结论 ---` **必填**：`通过` / `未通过` / `有画面` / `无画面` 等。
- 粘贴时**紧挨命令下方**即可；不必删上方命令行（保留便于对照）。

### 5.4 为何合并成单块（用户反馈）

| 旧方式 | 问题 |
|--------|------|
| 上方 `bash` 命令块 + 下方 distant「结果」块 | 上下距离远，频繁翻页，易贴错步骤 |

| 新方式 | 好处 |
|--------|------|
| 命令 → 结果 → 结论 **同一块** | 执行完向下滚动一点即粘贴，一步一处 |

### 5.5 异常情况处理

遇到以下情况时，**不必强行等待完整输出**，在 `# --- 结论 ---` 写明后触发分析：

| 异常类型                    | 结论行写法                                   | Agent 处置                                 |
| --------------------------- | -------------------------------------------- | ------------------------------------------ |
| 命令执行超时（>30s 无响应） | `命令超时，等待 XXs 无输出`                  | 判断是否需要 Ctrl+C 重试，或换命令         |
| 板子掉线 / SSH 断连         | `SSH 断连，板子已重启`                       | 确认板子恢复方式，决定是否重跑当步         |
| 输出不完整（被截断）        | `输出被截断，已贴前 N 行`                    | 根据已有输出分析，必要时要求补充关键行     |
| 命令报错退出                | `命令报错，exit code: N`                     | 分析错误原因，给出修复建议后才推进         |
| 结果与预期完全不符          | `结果异常，与通过标准不符`                   | 更新「已知事实」，重新分析根因，不强行继续 |

**通用原则**：只要 `# --- 结论 ---` 写清楚，Agent 就应能给出下一步处置，不需要完美日志。

## 6. 聊天触发语（收敛为统一格式）

所有触发语统一使用以下格式，消除歧义：

```
读 DEBUG_SESSION，<动作> ISSUE-00N Step M
```

| 意图         | 标准触发语                                         | 说明                               |
| ------------ | -------------------------------------------------- | ---------------------------------- |
| 新对话启动   | `读 DEBUG_SESSION，查看当前状态`                   | 上传 INDEX + 当前 ISSUE 文件后发送 |
| 执行前有疑问 | `读 DEBUG_SESSION，解答 ISSUE-00N Step M 问题`     | 已在文档里填好问题再发             |
| 执行完要分析 | `读 DEBUG_SESSION，分析 ISSUE-00N Step M 结果`     | 已在文档里填好结果再发             |
| 遇到异常     | `读 DEBUG_SESSION，分析 ISSUE-00N Step M 异常结果` | 已在 `# --- 结论 ---` 按 §5.5 填写   |
| 新建 ISSUE   | `新建 ISSUE-00N：<标题>`                           | Agent 在文档末尾按模板追加         |

> **不再使用**「请分析」「读 DEBUG_SESSION 继续」等宽泛说法，避免 AI 猜测意图。

## 7. Agent 行为要求（协作预期）

1. **新对话必须等用户上传文件**，未收到文件时拒绝开始分析（见 §0）。
2. **分析前必须先读** 上传的 ISSUE 文件，不凭聊天记忆推断。
3. **通过标准为空时，不得标 `✅ 可以执行了`**，必须先要求用户或自行补填。
4. **一次只展开下一步**；上一步未通过时，不让用户做后续无效步骤。
5. **异常结果不强行推进**；根据 §5.5 的异常类型给出处置建议，等用户确认。
6. **解答写在 markdown 里**，聊天里简短总结即可。
7. **静态知识**放 `CAMERA_VERIFY.md` 等；**当前会话进展**只记在对应 ISSUE 文件里。
8. 每步命令必须写清 **目的** 和 **通过标准**（量化优先，如「码率 ≥ 800000」优于「有码率」）。
9. 每步只使用 **一个** `shell` 块（命令 + 结果 + 结论），**不得**拆成远距离双代码块（§5.1）。
10. **更新 DEBUG_SESSION 日期前必须先执行 §12**，写入 INDEX「最后更新」与 ISSUE「变更记录」。

## 12. 日期与时间（变更记录必填）

DEBUG_SESSION 里的 **`最后更新`**、**`变更记录`** 必须使用**当次编辑时**从互联网取得的日期，**禁止**：

- 使用训练数据 / 聊天上下文里的「今天」
- 复制粘贴旧条目日期
- 不写日期

### 12.1 Agent 必做（每次改 INDEX / ISSUE 时）

1. **先取日期**（必须执行脚本，按顺序尝试多个互联网源；**禁止**手写或猜日期）：

```bash
sh scripts/debug_session_date.sh
# stdout → YYYY-MM-DD；stderr → 来源（worldtimeapi / timeapi.io / HTTP Date / local fallback）
```

脚本内部顺序：`worldtimeapi.org` → `timeapi.io` → 常见站点 HTTP `Date` 头（转 Asia/Shanghai）→ 本地 `TZ=Asia/Shanghai date`（仅网络全失败时，变更记录须注明 `local fallback`）。

2. 用得到的 `YYYY-MM-DD` 更新：
   - `DEBUG_SESSION/INDEX.md` → `**最后更新：** YYYY-MM-DD（…）`
   - 对应 ISSUE 文件 → `变更记录` **追加一行**（不修改历史行的日期）

3. 时区：**Asia/Shanghai**（与开发板/USB 调试时区一致）；API 不可用时用 `TZ=Asia/Shanghai date +%Y-%m-%d` 并在变更记录注明 `local fallback`。

### 12.2 格式示例

```markdown
**最后更新：** 2026-06-08（Step 8 收口）

| 2026-06-08 | Step 8 未通过；ISSUE 收口 |
```

历史条目日期**保持不变**（记录真实发生日）；仅**新追加**的行用当次查询日期。

## 8. 用户行为要求（协作预期）

1. **每次新对话，先上传 `INDEX.md` + 当前 ISSUE 文件**，再发触发语。
2. 只看 **⏳ 待执行** 或 **✅ 可以执行了** 的那一步。
3. 有疑问 → 先写「执行前问答」→ 等 Agent 标可执行 → 再跑命令。
4. 跑完 → 在当步 `shell` 块 `# --- 结果 ---` **紧挨命令下方**粘贴（含 `# --- 结论 ---`）→ 发标准触发语。
5. 遇到异常 → 在 `# --- 结论 ---` 按 §5.5 填写 → 发异常触发语，不要自行判断继续。
6. 不提前执行「预留 / Step N+1」里的内容。
7. 新问题开新 ISSUE 文件，避免和旧 issue 搅在一起；同步更新 `INDEX.md`。
